## What this repo is

`su-search-dev` is the B-side (development) frontend application for Seattle University's search system. It is a Next.js app deployed on Vercel. The system as a whole has four currently-deployed apps:

- `su-search` — A-side frontend (production)
- `su-search-dev` — B-side frontend (this repo)
- `funnelback-proxy` — A-side proxy (production)
- `funnelback-proxy-dev` — B-side proxy (sibling repo)

The frontend is the public-facing layer that T4 (the CMS managing seattleu.edu) imports scripts from. The proxy is the intermediary between the frontend and Funnelback/Squiz, holding credentials and handling the upstream search calls.

## Key architectural facts Claude Code should know

These facts affect how code in this repo should be read and summarized. They are the product of audit work that has already been done; they are not inferences to re-derive. The canonical audit documents in `audit/` (see "Audit documents available in this repo" below) hold the evidence; this section holds the headlines.

**A/B topology is being retired in the rebuild.** The current A/B mirror pattern is not carried forward. Any reasoning about architecture should not treat A/B as a design to preserve. It is a historical choice being undone.

**"One great app vs. front-back pair" is an open architectural question.** The rebuild may merge frontend and proxy into a single Next.js app, or keep them as two separate apps minus the A/B duplication. Working lean as of v17: Reading B (one unified app), but not yet committed. Analyses that depend on this fork should flag the dependency rather than assume either answer.

**Four live content-type renderers, not nine.** The Funnelback `partial.ftl` template imports nine result-type templates (news, law, programs, people, video, facebook, events, twitter, instagram), but only the first four are rendered in the Seattle University UX. Phase E (2026-05-14) confirmed via Vic that no crawler feeds the five inactive content types — they are structurally inert because the data axis underneath is empty, not because the frontend filters them out. Reason about content-type specialization in terms of the four live renderers plus basic search and autocomplete.

**Renderer dispatch is profile-config-driven.** Per Phase E, `result_list.Result` in `result_list.ftl` is the dispatch entry point. The dispatch key is `stencils.template.result.${result.collection}`; falls back through `result.gscopesSet` matched against `stencils.template.result.${gscope}`; falls back to the `results` namespace's `Result` macro (in `results.ftl`); errors otherwise. The four active per-renderer templates (`results.news.ftl`, `results.law.ftl`, `results.programs.ftl`, `results.people.ftl`) plus the `results.ftl` fallback all emit a shared `.listing-item` BEM envelope with per-renderer modifier class suffix (`--generic`, `--course` on news — copy-paste residue per below, `--program`, `--people`) and per-renderer `data-class` suffix (`genericData`, `lawData`, `newsData`, `programData`, `peopleData`). Per Phase D D7, the `dataListing` class and the per-type `*Data` suffixes are preserved verbatim by the rebuild pending Phase B consumer enumeration — the camelCase paired generic-plus-per-type pattern suggests JS hook selectors more than visual or ARIA semantics, but consumer enumeration is the binding evidence and was not performed in Phase E. Do not treat `dataListing` as redundant-with-`.listing-item` or as a cleanup target; that framing was Phase E speculation, corrected by D7.

**Visual-design-immutability is a binding constraint (v18 anchor).** The rebuild makes no changes to the visual design. This covers CSS class names referenced by the stylesheet, markup envelopes whose styling depends on exact class assignment, the SVG sprite contents, and chrome whose appearance is locked even where its behavior is open. The working default on every visual-design-adjacent surface is preserve-verbatim; any cleanup that touches the visual surface needs explicit justification rather than being default-on. Several findings that earlier read as cleanup candidates are constraints under this anchor — the `listing-item--course` modifier on news (below), the lowercase `promoted` curator label, and the cross-renderer envelope shape.
>
> **Stylesheet-as-binding-contract (established 2026-06-02; pending v19 formalization).** The existing Funnelback stylesheet lives in T4, is dedicated to Funnelback styles, and is a binding restriction: everything the rebuild builds must align with it. This is the active form of visual-design-immutability — not just "do not change the appearance" but "the stylesheet is fixed, and the rebuild's markup must emit the class and selector contracts it already expects." For any styled class, the rebuild emits what the stylesheet targets rather than renaming it. Practical effect when reading result-card and envelope markup: a class with no FE-JS consumer but a stylesheet selector is emit-and-align, not a cleanup candidate.
>
> **Preserve-DOM-contract-by-default-pending-consumer-enumeration (v18 anchor).** Every class name, data attribute, ARIA attribute, and selectable element with no audit-confirmed consumer is preserved by default. The rebuild does not remove a contract surface on the assumption that it has no consumers — it removes only after consumer enumeration confirms the absence. This is the principle that converts most Phase B handoff items from cleanup-decisions into documentary consumer-enumeration: the question is "what consumes this," not "can we cut this."

**Autocomplete: two consumers, one shared backbone.** Header autocomplete (in `public/integration.js`, on every page where `#search-input` exists) and search-page autocomplete (in `public/search-page-autocomplete.js`, attaching site-wide despite the file name but only doing useful work on `/search-test/`) are two separate consumers. The header consumer renders only the general bucket from the response; the search-page consumer renders all three in a three-column layout. The endpoint is one tier with three internal legs in the default branch.

**Two `performSearch` implementations exist and are wire-distinguishable.** `integration.js`'s private `performSearch` (line 899, with `window.performSearch` wrapper at line 1280) sends `form=partial` explicitly; `search-page-autocomplete.js`'s local `performSearch` (line 666) does not. The May 6 listener audit's puzzle 3 confirmed via static analysis that both search-page click-handler call sites (L550 inside the staff/program inner-`<a>` setTimeout, L564 in the main branch) lexically resolve to the local L666 function — function declarations are hoisted to the module scope and found before identifier resolution reaches `window.performSearch`. Regardless of which fires in current production, the rebuild collapses these to one canonical dispatcher with `form` as an explicit parameter.

**`tabs-manager.js` patches two window globals at module-load time; `destroy()` is bounded-today non-functional.** During its `initialize()` (called from the constructor at L40, which itself runs inside the dynamic-import resolution chain at `core-search-manager.js:249` → `new ModuleClass(this)` at L253), tabs-manager replaces `window.performSearch` and `window.updateSearchUrl` with bound wrappers (`tabs-manager.js:54`, `:65`). Because `core-search-manager.js` is a deferred module loaded by `search-index.js`, the wrapping happens after `integration.js`'s DOMContentLoaded handler completes — any code that captured those globals earlier holds a reference to the un-wrapped function. Separately, `destroy()` at `tabs-manager.js:543–559` calls `removeEventListener` with `this.handleTabClick.bind(this)`; `bind` produces a new function reference per call, so the removal does not match the originally-attached listener. Both observations are bounded today (the manager is constructed once per page session and `destroy()` is never invoked anywhere in the codebase). The rebuild establishes explicit dispatch rather than global-function patching, plus a real lifecycle model with reference-stable handler identities.

**Four-request fan-out on tab clicks is resolved — composition of two attachment paths.** Was tracked as open through the listener audit and tabs-manager audit; closed 2026-05-11 by the network recap re-examination addendum. The mechanism is a deferred-DOM-availability defect: `installTabClickHandlers()` attaches a document-level fallback listener at module-init (when `findTabContainers()` returns empty), then M1-driven `handleDomChanges()` later attaches per-container listeners on three ancestor elements. Both attachment paths' listeners coexist at runtime because they originate from different lifecycle moments. Every tab click bubbles through all four, firing `handleTabClick` four times. The fix in the rebuild is single-attachment-strategy per concern, with provisional fallback listeners removed once their preconditions are met.

**Tab ↔ facet coupling is server-driven, not FE-orchestrated.** The Funnelback partial template scopes facets to the active tab via `stencils.tabs.facets.${selected}` — that is a *backend* mechanism. The FE has zero references to `stencils.tabs.*` across `public/`, `pages/`, or `lib/`. When a tab is clicked, `tabs-manager.js`'s `loadTabContent` (L351) fetches the new tab's URL and replaces `#results` innerHTML with the server's response, which already contains the correct facet block. `facets-manager.js` re-binds against the new DOM via the M1 MutationObserver cascade in `core-search-manager.js` (L332). There is no FE-side cross-module dispatch from tabs-manager to facets-manager — no direct call, no shared-state write, no custom event. The two managers are coupled through the rendered DOM, not through control flow.

**Pre-render helpers live in the autocomplete file.** `checkForPreRenderedContent` (search-page-autocomplete.js:168) and `displayPreRenderedResults` (line 290) are exposed globally at lines 1133-1134 for `integration.js`'s `processUrlParameters` to call. The file owns two responsibilities — autocomplete plus pre-render display — that the rebuild may want to split.

It represents an incomplete migration (the planned consolidation of the four-script approach). Safe to note as removal candidate for the rebuild; not to be modified now.

**`integration.js`'s `setOriginalQuery` consumer is dead code.** `integration.js:696–702` checks `typeof window.SearchManager.setOriginalQuery === "function"` and calls it if present, otherwise falls through to a direct property write at L705. A grep across `public/`, `pages/`, and `lib/` returns zero definitions for `setOriginalQuery` — no per-feature manager defines it; the `SearchManager` class itself does not declare it. **`core-search-manager.js` only initializes on URL paths containing "search".** It is a search-results-page orchestrator, not a site-wide thing. Autocomplete on non-search pages is powered by `search-page-autocomplete.js` directly.

## Load tier reality

The T4 PageLayout imports four scripts in three load tiers:

1. **High priority** (`fetchpriority="high"`): `SessionService.js`
2. **Synchronous blocking** (no load attribute): `search-page-autocomplete.js`
3. **Deferred**: `integration.js` (`defer`), `search-index.js` (`type="module"`, deferred by default)

Tier 2 being synchronous is a fact, not necessarily a commitment. Whether it needs to be synchronous is a question for the rebuild's bundle-shape review.

Beyond the four HTML-loaded scripts, six per-feature managers (`tabs-manager.js`, `facets-manager.js`, `pagination-manager.js`, `spelling-manager.js`, `analytics-manager.js`, `collapse-manager.js`) load dynamically via `import()` at `core-search-manager.js:249`. They are not in the HTML. The full per-page-navigation chain is: HTML → `search-index.js:14` imports `core-search-manager.js` → `core-search-manager.js:249` imports each per-feature manager. The May 6 listener audit's puzzle 7 confirmed this is the sole load path for the per-feature managers; there is no double-import and no double-initialization.

## Cache implementation

`lib/cache.ts` implements the frontend cache. The cache audit (`audit/fe-cache-audit-2026-05-04.md`) is the canonical reference. Headlines: a narrow `getCachedData` / `setCachedData` get/set surface that fails safely on Redis errors; five TTL tiers aligned to crawl cadence; an in-memory `queryPopularity` map whose TTL-tuning half is alive on exactly one call path (search miss-fetch) but bounded by serverless container warmth, and whose metrics-rollup half is dead with no consumer. Five call sites with four different normalization patterns and five different TTL behaviors. The env var names (`su_search_dev_012026_KV_URL`, etc.) embed a timestamp and serve as the system's actual cache-flush mechanism — renaming the env var to repoint at a fresh KV store is the only effective full-flush practice. The rebuild will use stable env var names with explicit versioning if needed.

**Suggestions caching is structurally distinct from search caching.** Two layers of distinction: the `suggestions:` cache-key prefix in `pages/api/suggestions.ts:57` bypasses the popularity-tier surface in `lib/cache.ts` (the prefix sniff matches only `search:` and `tab:`); and `pages/api/suggestions.ts:71` does not pass `cacheAware: true` to `createApiClient`, so suggestions does not assert the X-Cache-Aware contract on the upstream proxy that `search.ts` and `pre-render.ts` do. Whether this dual divergence is intentional or accidental is rebuild design space.

**The "tab-cache namespace" is a misnomer.** The `tab:` cache namespace in `lib/cache.ts` is *not* populated by tab-click requests at all. Tab clicks bypass `lib/cache.ts` entirely. The `tab:` namespace is populated by form=partial requests routed through `pages/api/search.ts:127–145`'s substring-inference fallback — i.e., search-form submission flows where URL substring matching classifies the request as tab-like. The rebuild should make tab-navigation an explicit request type with metadata rather than inferred from URL substrings.

**The partial response is the cache *output*, not a cache *consumer*.** Per Phase E and v17. The FTL templates that compose the Funnelback partial response do not participate in any caching decision — caching decisions live downstream of `partial.ftl`'s rendering, at the FE-server (`lib/cache.ts`) and proxy-side cache layers.
